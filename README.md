## Project Overview

This project was completed as part of a **Data Science Internship** (Duration: 2 weeks). The objective is to build, tune, and evaluate two powerful ensemble machine learning models — **XGBoost** and **Random Forest** — to solve a real-world HR classification problem: predicting which employees are likely to leave the company.

### Objectives

- Build and compare XGBoost and Random Forest classifiers on the same dataset
- Handle class imbalance using SMOTE with leakage-free cross-validation
- Perform hyperparameter tuning using `RandomizedSearchCV` with identical settings for both models to ensure a fair comparison
- Analyse feature importance using Gini (RF), Gain (XGBoost), and Permutation methods
- Deploy an interactive Streamlit app that runs both models live and shows predictions side by side

### Why this comparison matters

Random Forest and XGBoost represent two fundamentally different ensemble strategies — **bagging** vs **boosting**. RF builds many independent trees and averages their votes. XGBoost builds trees sequentially, with each tree correcting the errors of the previous one. On imbalanced HR data, these strategies behave differently, and this project quantifies exactly how.

## Methodology

### 1. Exploratory Data Analysis

- Target distribution analysis — confirmed 84:16 class imbalance
- Numerical feature distributions with histograms
- Key features plotted by attrition status (Yes vs No)
- Attrition rates per categorical feature (Department, JobRole, MaritalStatus, OverTime, etc.)
- Full correlation heatmap with top features correlated with attrition
- Outlier detection using IQR method — kept outliers as tree-based models are robust to them

### 2. Data Cleaning & Preprocessing

- Dropped zero-variance columns: `EmployeeCount`, `StandardHours`, `Over18`, `EmployeeNumber`
- Binary label encoding: `Gender` (Male=1, Female=0), `OverTime` (Yes=1, No=0)
- One-hot encoding for nominal columns: `BusinessTravel`, `Department`, `EducationField`, `JobRole`, `MaritalStatus`
- Target encoding: `Attrition` → Yes=1, No=0
- Final feature set: **44 columns**

### 3. Class Imbalance — SMOTE

Applied **SMOTE** (Synthetic Minority Oversampling Technique) on the training set only, after the train/test split:

- Original training set: 1,176 samples (986 No / 190 Yes)
- After SMOTE: 1,972 samples (986 No / 986 Yes)
- Test set: 294 samples — **never oversampled**, used for honest evaluation

### 4. Leakage-free Cross-Validation — ImbPipeline

A critical methodological detail: SMOTE must be applied **inside** each CV fold, not before cross-validation. Applying SMOTE before CV causes synthetic copies of the same original point to appear in both the train and validation folds, inflating CV scores by 15–20 AUC points.

Solution: `imblearn.pipeline.Pipeline` (ImbPipeline) wraps SMOTE inside the pipeline so it is applied only to the training portion of each fold:

```python
from imblearn.pipeline import Pipeline as ImbPipeline

pipeline = ImbPipeline([
    ('smote', SMOTE(random_state=42)),
    ('model', RandomForestClassifier(...))
])
cross_val_score(pipeline, X_train_orig, y_train_orig, cv=cv, scoring='roc_auc')
```

Cross-validation was always performed on `X_train_orig` (original unbalanced data). Training the final model used `X_train` (SMOTE-balanced).

### 5. Hyperparameter Tuning

Both models used **identical search settings** for a fair comparison:

| Setting | Value |
|---|---|
| Method | `RandomizedSearchCV` |
| Iterations | 80 |
| CV folds | 5 (Stratified) |
| Scoring metric | `roc_auc` |
| Random seed | 42 |

**Random Forest search space:**

| Parameter | Values |
|---|---|
| `n_estimators` | 100–600 |
| `max_depth` | None, 5, 10, 15, 20, 25 |
| `min_samples_split` | 2–20 |
| `min_samples_leaf` | 1–10 |
| `max_features` | sqrt, log2, 0.5, 0.7 |
| `class_weight` | balanced, balanced_subsample, None |
| `criterion` | gini, entropy |

**XGBoost search space:**

| Parameter | Values |
|---|---|
| `n_estimators` | 100–600 |
| `max_depth` | 3, 4, 5, 6, 8, 10 |
| `learning_rate` | 0.01–0.30 |
| `subsample` | 0.6–1.0 |
| `colsample_bytree` | 0.6–1.0 |
| `gamma` | 0–0.5 |
| `min_child_weight` | 1–10 |
| `reg_alpha` | 0–1 |
| `reg_lambda` | 0.5–2.5 |
| `scale_pos_weight` | 1, class_ratio |

### 6. Threshold Optimisation

Both models were evaluated by sweeping all thresholds from 0.01 to 0.99 and selecting the threshold that maximises F1. This is important for imbalanced classification where the default 0.5 threshold typically under-detects the minority class.

## Results & Findings

### Performance Comparison — Default Threshold (0.5)

| Metric | Random Forest | XGBoost | Winner |
|---|---|---|---|
| Accuracy | 0.8061 | 0.8469 | XGBoost |
| Precision | 0.3750 | 0.5294 | XGBoost |
| Recall | 0.3191 | 0.3830 | XGBoost |
| F1-Score | 0.3448 | 0.4444 | XGBoost |
| ROC-AUC | 0.7338 | 0.7708 | XGBoost |
| PR-AUC | 0.3611 | 0.4345 | XGBoost |


### Key Takeaways

- **Both models benefit significantly from threshold tuning.** Lowering the threshold from 0.5 to the optimal value (typically 0.30–0.40) substantially improves recall — meaning more at-risk employees are correctly identified.

- **SMOTE + ImbPipeline is critical.** Without leakage-free CV, cross-validation scores are artificially inflated by 15–20 AUC points compared to real test performance.

- **XGBoost tends to concentrate importance** on a smaller set of high-gain features, while Random Forest distributes importance more broadly across the feature set.

- **Both models consistently identify the same core predictors** — features like OverTime, MonthlyIncome, StockOptionLevel, Age, and JobSatisfaction appear in the top 15 of both models.

- **The optimal threshold matters more than which model you choose** for HR use cases. At the optimal threshold, recall for the attrition class improves dramatically (0.32 → 0.70 typical) at the cost of some precision — which is the right trade-off for retention programs where missing an at-risk employee is more costly than a false alarm.
